# Throw In - 未実装（今後の開発）機能・非機能要件定義書

本ドキュメントは、「Throw In」アプリケーションにおいて、現在はUIプロトタイプ（モックアップ）に留まっており、今後バックエンドや外部APIを組み込んで「本番稼働」させるために実装が必要な要件を整理したものです。データベース・バックエンド基盤には **Supabase** を採用することを前提としています。

## 1. 機能要件 (Functional Requirements)

### 1.1. AI・データ処理パイプライン連携（コア機能）
*   **Web記事（URL）の自動スクレイピング・解析**: 入力されたURLから本文、タイトル、OGP画像（アイキャッチ）を抽出し、データベースへ保存するバックエンドワーカー。
*   **ファイル解析（OCR・テキスト抽出）**: PDFや画像がアップロードされた際、文字情報を読み取る（Google Cloud Vision API または Document AIなど）。
*   **動画の解釈**: YouTube URLなどから字幕（トランスクリプト）を取得し、テキスト化する処理。
*   **AI要約と構造化（OpenAI API連携）**: 抽出した長文テキストを、OpenAI API（モデル：`GPT-5.4-nano`）を用いて自動的に「要約（数行）」「重要なポイント抽出」「自動タグ生成」に変換する処理。処理にはアプリ側で固定されたシステムプロンプトを使用する。
*   **AI API利用戦略（プレビュー版と正式版の仕様差異）**:
    *   **設定・プレビュー期間限定**: ユーザー設定画面等から「ユーザー自身のOpenAI APIキー」を入力して利用させる。
    *   **本リリース以降**: プレビュー期間限定でのAPIキー入力箇所を廃止。ユーザーの利用プラン（制限付き）に応じて、サーバー側で管理するAdmin専用のOpenAI APIキーを使用する形へと切り替える。
*   **バックグラウンド非同期処理**: AIの処理（取得・抽出・要約）には時間がかかるため、登録直後は「AI処理中」ステータスにし、裏側のジョブキューで処理・完了時にUIを更新する仕組み。

### 1.2. データベースと永続化（CRUDの実装）
*   **Zustandからの脱却**: 現在メモリ上で管理しているクリップデータを、実際のデータベース（Supabase PostgreSQL）へ移行。
*   **コレクション管理機能**: クリップをまとめる「コレクション（フォルダやプレイリストのような概念）」の作成、編集、削除機能（現在は空のUIのみ）。
*   **履歴管理（History）**: ユーザーが閲覧したクリップの順序をトラッキングし、履歴ページに表示する機能。
*   **ファイルのストレージ保存**: ユーザーがアップロードした画像・PDFをSupabase Storageへ安全に保存・配信する機能（最大10MB等のバリデーション含む）。

### 1.3. ユーザー認証とマルチテナント (Authentication)
*   **アカウント登録 / ログイン**: Google OAuth等を用いたセッション管理（Supabase Auth）。
*   **個人データの分離化**: 「自分のクリップ」のみを表示・検索・編集できるユーザーデータ分離の適用。
*   **ログアウト・アカウント削除**: ユーザー自身によるデータ全削除退会機能。

### 1.4. 高度な検索エンジン (Search)
*   **全文検索**: タイトルやタグだけでなく、抽出された長文テキスト（本文）を対象とした高速な検索（SupabaseのPostgreSQL全文検索機能や、Algoliaとの連携）。
*   **曖昧検索 / AI検索**: 「あの時のAIについての記事」といった自然言語での検索クエリに対し、Supabase Vector（pgvector）等を利用してクリップをヒットさせる機能。

### 1.5. その他の拡張機能
*   **PWA Web Share Target API（共有機能）**: iOS/Androidにて「共有メニュー」から直接Throw InアプリにURLを渡すためのバックエンド処理。
*   **レポート機能**: 日次/週次/月次において、キュレーションした傾向をAIが分析して提示するインサイト生成機能。
*   **アドミン機能（管理画面）**: サービスの利用状況、ジョブのエラー率、ユーザー管理を行うための実際の統計データ連携。

## 2. 非機能要件 (Non-Functional Requirements)

### 2.1. セキュリティ (Security & Privacy)
*   **アクセス制御（RLS）**: SupabaseのRow Level Security (RLS) を用いた強固なアクセス制御（自分以外のデータを読み書きできないようにする）。
*   **SSRF対策**: サーバー側でユーザー指定のURLをスクレイピングする際、社内ネットワークや無効なIPに対する不正なリクエストを防ぐバリデーション。
*   **APIキーの隠蔽・暗号化**: プレビュー版においてユーザーが入力したOpenAI APIキーを安全に保持する仕組み。また正式版におけるAdmin共有シークレットキーを、クライアントに露出させずNext.js API Routes等でのみ実行する設計。

### 2.2. パフォーマンスとスケーラビリティ (Performance & Scalability)
*   **ページネーション・無限スクロール**: クリップが1000件、10000件に増えた際にブラウザのリソースを枯渇させないための、遅延ロード・カーソルベースページネーションの実装。
*   **画像最適化（Image Optimization）**: OGP画像やアップロード画像のNext.js `<Image>` またはCDNでのリサイズとキャッシュ。
*   **ジョブキューのスケーラビリティ**: 一度に大量のURLやPDFが投げ込まれても詰まらないためのタスクキューシステムの構築。

### 2.3. 信頼性と可用性 (Reliability & Offline)
*   **エラーハンドリング・リトライ**: URLが404だった場合やAIサーバーが一時的にダウンしている際に、ジョブをリトライする仕組みと、ユーザーフレンドリーなエラーUIの表示。
*   **オフラインサポート（PWA強化）**: メトロの中や通信制限環境下でも、キャッシュされたライブラリを閲覧でき、オフライン状態でのメモ作成をオンライン復帰時にバックグラウンド同期する機能。

## 3. 実装に向けたフェーズロードマップ
（※詳細な進捗管理は `docs/tasks.md` で行います）

*   **フェーズ 1: 基盤環境と認証のセットアップ (Supabase)**
    *   Supabaseプロジェクトの初期設定とデータベーススキーマの構築。
    *   Supabase Authを用いたユーザー認証の導入とRLSの適用。
*   **フェーズ 2: コアCRUDの実装とインメモリ状態からの移行**
    *   ZustandのモックデータをSupabase連携へ差し替え。
    *   ファイルのストレージアップロード（Supabase Storage）の実装。
*   **フェーズ 3: データパイプラインとAI連携 (コア機能)**
    *   URLからのデータ抽出（スクレイピング）、ファイル解析。
    *   ユーザー自身のAPIキーと OpenAI API (GPT-5.4-nano) を用いた自動要約とタグ付けパイプラインの開通。
*   **フェーズ 4: 高度な検索とUX向上**
    *   pgvectorや全文検索エンジンの統合。
    *   無限スクロール等のパフォーマンスチューニングとレポート機能。
*   **フェーズ 5: 正式リリースに向けたAI基盤の移行**
    *   ユーザー入力によるAPIキー利用設定の廃止。
    *   利用プランに応じたAdmin管理のAPI利用（回数・容量制限付き）への切り替え。
