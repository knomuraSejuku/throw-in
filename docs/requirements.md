# Throw In - 実装済み機能・非機能要件定義書

本ドキュメントは、現在までに実装・確認されている「Throw In」アプリケーションの機能要件および非機能要件をまとめたものです。

## 1. アプリケーション概要
情報のノイズから離れ、本質を探究するための「デジタルキュレーター」アプリ。ウェブ記事、ドキュメント、動画、個人的な日記など、あらゆる情報を1箇所に集約し、AIによる要約やテキスト抽出、タグ付けを通じて情報を整理・活用します。

## 2. 機能要件 (Functional Requirements)

### 2.1. ライブラリ画面（一覧・検索・絞り込み）
*   **ビューモード切り替え**:
    *   グリッドビュー（サムネイル・概要を強調するおしゃれなカード形式）
    *   リストビュー（1クリップ1行のコンパクトな表示）
    *   原典（Sourcebook）モード（表形式。元URL/ファイル名、タイプ、タグ、保存日などを一覧表示）
*   **クイックフィルター**:
    *   「未読のみ」トグルボタンによるライブラリ内検索からの即時絞り込み。
*   **サイドバーナビゲーションによる精細な絞り込み**:
    *   ビューによる絞り込み（すべてのクリップ、ブックマーク、履歴）
    *   メディアタイプによる絞り込み（記事、動画、画像、ドキュメント、日記・メモ）
    *   よく使うタグによる絞り込み（#Design, #Tech 等）

### 2.2. クリップ詳細画面
*   **ヘッダーのスクロール追従（Sticky Header）**: 画面を下へスクロールした際、ドメイン、日付、メディアタイプ、タイトルを含むコンパクトなヘッダーが上部に固定表示され、横スクロールに対応。
*   **コンテンツ表示**:
    *   元サイトURLや情報源への外部リンク遷移機能
    *   サムネイル表示
    *   AI要約の表示パネル
    *   本文の抽出プレビュー表示（長文の場合は「全文を表示 / 折りたたむ」でトグル開閉可能）
*   **マイメモ（パーソナルメモ）機能**:
    *   ユーザー固有のメモを表示。
    *   「編集」ボタンからその場でテキストエリアに切り替わり、追加・修正して状態を保存する機能（Storeに連動）。
*   **アクション**:
    *   ブックマークの登録・解除（トグル機能・Store連動）
    *   未読アイコンの表示と、詳細画面へのアクセス時における自動既読（Read）設定化

### 2.3. クリップの新規追加（データ登録）
*   **入力モードの切り替え**:
    *   URL保存（ウェブ記事向け）
    *   アップロード（画像、PDFドキュメント向け）
    *   日記作成（ローカルのメモ向け。デフォルトでシステムの「今日の日付 YYYY-MM-DD」が入力済み）
*   **メタ情報の設定と保存**:
    *   カスタムタイトル、本文/メモの入力、タグの設定機能。
    *   「保存する」ボタンにより、ZustandのグローバルStoreにデータが即時追加され、ライブラリ画面に新しいクリップとして反映される。

## 3. 非機能要件 (Non-Functional Requirements)

### 3.1. デザイン・UI/UX
*   **デザイントレンド**: 「Fluid Intelligence（流体知能）」をテーマに、透明感（Backdrop Blur）、大胆な角丸（Rounded Corners・`rounded-[32px]`等）、トナルサーフェス（階層に基づく背景色）を駆使したモダンUI。
*   **カラースキーム**: プライマリ、セカンダリ、ターシャリなど、視覚的に意味のあるカラーパレットをCSS Variablesで定義。
*   **カスタムスクロールバー**: WebブラウザやOSのデフォルトスクロールバーを細く角丸なデザインに上書きし、アプリ全体のアピアランスと統一。
*   **レスポンシブデザイン（モバイルファースト視点のハイブリッド）**:
    *   スマートフォンなどモバイル環境では画面下部固定のボトムナビゲーションバーを表示。
    *   PC/タブレット環境では左側の固定サイドバーを表示し、効率的なIA（情報アーキテクチャ）を提供。
    *   iOS端末等のセーフエリア（Notch、Home Indicator）を考慮した余白調整（`pb-safe` クラス）。

### 3.2. フロントエンドアーキテクチャ・技術スタック
*   **フレームワーク**: Next.js 15 (App Router), React.
*   **スタイリング**: Tailwind CSS v4（ユーティリティファースト、高度なテーマカスタマイズ）。
*   **状態管理（State Management）**: `zustand` を利用したクライアントサイド・グローバルステート管理。モックデータの保持、追加・更新ロジックを `lib/store.ts` に集約。
*   **アイコンシステム**: `lucide-react` を用いた一貫したベクターアセット。

### 3.3. パフォーマンスとインタラクション
*   **アニメーションとトランジション**: Tailwindの `transition` や `clsx` による柔軟なクラス制御を通じ、リストのホバー時、ボタンのアクティブ時、ヘッダーの出現時等に滑らかな動きを実装。
*   **状態の非同期同期**: 現状はフロントエンド（メモリ内）での高速なStore更新によるゼロレイテンシのUIフィードバックを実現。

## 4. 今後の開発課題・未実装領域（ネクストステップ）
1.  **AI連携のバックエンド処理**: OpenAI API (GPT-5.4-nano) を用いた、URLやPDFからの実テキスト抽出、自動構造化、タグ推論、AI要約の自動生成のパイプライン構築。
    *   *運用方針*: プレビュー期間中は、ユーザー自身が設定画面で自分用のAPIキーを入力する形式とします。正式リリース時はこの入力を廃止し、利用プランに応じてAdmin APIキーを利用する仕組みへと移行します。
2.  **データベース永続化と認証**: Supabase等のBaaS・データベース基盤を導入し、Zustandのモックデータを実際のデータベース（Supabase PostgreSQL）への書き込み/読み込みへリプレイス。Supabase Auth等でのユーザーログイン機能。
3.  **検索エンジンの構築**: 高速な全文検索、高度なフィルタリングクエリのバックエンド実行（Algolia等、もしくはSupabase PostgreSQLの全文検索やpgvectorの活用）。
4.  **コレクション管理機能**: クリップを任意の「コレクション」に束ねる機能の実装（現在はUIモックのみ）。
